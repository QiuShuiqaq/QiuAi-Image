# QiuAi Prompt Tag Library Design

## 背景

当前 `QiuAi` 已有两类提示词能力：

- `按钮提示词`：用于 `商品主图 / 详情图 / 细节图 / 尺寸图 / 白底图 / 颜色图` 等固定图片类型提示词。
- `自定义提示词模板`：用于整段提示词的本地存储与按需导入。

但 `套图设计` 与 `套图生成` 的 `全局主提示词` 仍是纯文本输入。对普通用户来说，这种方式有三个问题：

- 不易复用，常见风格片段需要反复手输。
- 不易规避风险词，用户不知道哪些词会提高“保留原图”概率。
- 不易维护，模板和“碎片化提示词片段”混在一起，后续扩展会越来越乱。

本次设计在不破坏现有模板体系的前提下，引入独立的 `标签库`，让 `套图设计` 与 `套图生成` 的 `全局主提示词` 改为通过标签拼接生成，同时在命中高风险词时走顶部消息提示。

## 目标

- 为 `套图设计` 与 `套图生成` 增加标签化的 `全局主提示词` 组装能力。
- 在 `提示词库` 页面新增独立 `标签库` 管理区，支持标签分类与标签内容维护。
- 保留现有 `按钮提示词` 与 `自定义提示词模板` 逻辑，不做破坏性迁移。
- 对高风险提示词做即时检测，并通过顶部消息提示反馈给用户。

## 非目标

- 不改 `单图测试` 与 `单图设计` 的提示词输入方式。
- 不阻断用户提交任务，不加入强制弹窗确认。
- 不改后端任务接口的 `globalPrompt` 字段协议。
- 不处理标签拖拽排序，初版按用户选择顺序拼接。

## 设计概览

### 1. 概念边界

系统内同时保留三套不同职责的数据：

- `按钮提示词`
  用于图片类型按钮联动，例如主图、详情图、细节图等。
- `自定义提示词模板`
  用于整段提示词复用，仍服务于“导入按钮”逻辑。
- `标签`
  用于 `全局主提示词` 的碎片化拼接，例如“高清”“白底”“商业质感”“换背景”等。

三者职责严格分离：

- 图片类型相关，走按钮提示词。
- 整段导入，走自定义提示词模板。
- 全局风格拼接，走标签库。

### 2. 影响模块

- `renderer/src/components/PromptLibraryPanel.vue`
- `renderer/src/components/ParameterSettingsPanel.vue`
- `renderer/src/App.vue`
- `renderer/src/services/desktopBridge.js`
- `main/src/services/promptTemplateStoreService.js`
- 新增独立标签存储服务与 IPC
- 与 `studioWorkspaceService` 的草稿持久化逻辑联动

## UI 设计

### 1. 提示词库页面

现有 `提示词库` 页面继续保持三栏结构，但中间栏职责调整：

- 左栏：`按钮提示词 + 自定义提示词`
- 中栏：`标签库`
- 右栏：`提示词存储`

#### 左栏

保留当前两类内容：

- 固定按钮提示词可展示、编辑、保存。
- 自定义提示词模板可新建、编辑、删除。

这里不承载标签逻辑，避免模板和标签混用。

#### 中栏

改为 `标签库`，按分类展示：

- 每个分类显示分类名与该分类下的标签列表。
- 支持新建标签分类。
- 支持重命名标签分类。
- 支持删除空分类；非空分类删除时需先删除内部标签，避免误删整类内容。
- 支持在分类内新增标签。
- 支持编辑标签名称。
- 支持编辑标签实际拼接的提示词片段。
- 支持删除单个标签。

标签内容示例：

- 标签名称：`高清`
- 标签提示词：`高精度商业电商画面，主体清晰锐利，细节完整。`

#### 右栏

继续作为只读式存储总览区，但要把标签也纳入总览，便于用户统一查看本地沉淀内容。

排序规则：

- 仍按名称排序。
- 展示来源：`按钮提示词 / 自定义提示词 / 标签`

### 2. 参数设置页面

仅影响：

- `套图设计`
- `套图生成`

#### 全局主提示词区域

原来的 `textarea` 改为标签操作区：

- 显示 `已选标签`
- 显示 `清空标签` 按钮
- 显示 `导入标签` 或同等语义按钮
- 显示拼接后的只读预览文本

交互规则：

- 点击标签后加入已选标签列表。
- 重复点击同一标签不重复加入。
- 删除已选标签后，拼接结果实时更新。
- 初版按选择顺序拼接，不做拖拽排序。

拼接结果作为 `globalPrompt` 的实际提交值。

#### 旧草稿兼容显示

若加载到历史草稿时只有 `globalPrompt` 文本、没有 `selectedTagIds`：

- 保留历史文本可见。
- 页面标记为“历史文本模式”。
- 用户首次选择或删除标签后，切换到标签模式。
- 切换后以标签拼接结果为准。

这样可以避免旧草稿在升级后直接丢失。

## 数据设计

### 1. 保留现有模板存储

继续保留 `promptTemplates`：

- 固定按钮提示词
- 自定义提示词模板

不在这个存储里混入标签数据。

### 2. 新增标签库存储

新增独立本地存储键，例如：

- `promptTagCategories`

数据结构建议为：

```js
[
  {
    id: 'category-quality',
    name: '画质',
    tags: [
      {
        id: 'tag-hd',
        name: '高清',
        prompt: '高精度商业电商画面，主体清晰锐利，细节完整。'
      }
    ]
  }
]
```

这样可以保证：

- 分类与标签天然归属明确。
- 前端渲染结构直接对应数据结构。
- 后续新增分类排序、折叠、搜索都容易扩展。

### 3. 草稿结构

在 `套图设计 / 套图生成` 草稿中新增：

- `selectedGlobalTagIds`
- 可选 `legacyGlobalPrompt`

兼容策略：

- 新草稿以 `selectedGlobalTagIds` 为主。
- 老草稿若只有 `globalPrompt` 文本，则保存为历史文本显示来源。

提交前前端计算：

```text
globalPrompt = selected tag prompt 按选择顺序用换行拼接
```

后端仍然只接收 `globalPrompt` 字符串，不感知标签结构。

## 风险检测设计

### 1. 检测范围

风险检测覆盖三种来源：

- 已选标签拼接结果
- 用户导入的自定义提示词模板
- 标签库中被编辑后的标签内容

### 2. 风险等级

高风险词示例：

- `和原图一致`
- `保持原样`
- `不改动布局`
- `复刻原图`
- `完全一致`
- `不要变化`

中风险词示例：

- `尽量不变`
- `保留原图风格`
- `轻微修改`
- `只改一点`
- `背景不动`

### 3. 反馈方式

命中风险词时：

- 不阻断操作
- 不弹出确认框
- 走现有顶部消息提示

提示文案示例：

- `检测到高风险提示词“和原图一致”，可能导致保留原图或生成失败`

一次命中多个词时：

- 仅提示最高风险的一条
- 避免连续刷屏

### 4. 触发时机

以下操作发生时立即检测：

- 选择标签
- 删除标签
- 清空标签
- 导入自定义提示词模板
- 编辑标签内容并保存

## 提交流程

### 套图设计 / 套图生成

1. 用户在参数区选择多个标签。
2. 前端根据 `selectedGlobalTagIds` 解析出对应标签内容。
3. 按选择顺序拼接为最终 `globalPrompt`。
4. 若命中风险词，先给顶部消息提示。
5. 提交任务时，仍按原协议把字符串 `globalPrompt` 发给现有后端流程。

这样可以保证：

- 现有 `studioImageGenerationService` 无需因标签引入大改。
- 草稿持久化与任务提交保持兼容。

## 错误处理

- 标签分类名称为空时，不允许保存，并给顶部错误提示。
- 标签名称为空时，不允许保存，并给顶部错误提示。
- 标签提示词为空时，不允许保存，并给顶部错误提示。
- 引用不存在的标签 ID 时，提交前自动忽略失效标签，并给顶部警告提示。
- 删除标签后若草稿里仍引用该标签，加载草稿时自动清理失效 ID。

## 测试策略

### 1. 前端源测试

验证：

- `套图设计 / 套图生成` 的 `全局主提示词` 区域不再是主要文本输入方式。
- `提示词库` 中存在 `标签库`、`新建标签分类`、`已选标签`、`清空标签` 等关键文案或结构。

### 2. 前端行为测试

验证：

- 选择多个标签后拼接顺序正确。
- 重复点击同一标签不会重复加入。
- 删除标签后拼接结果同步变化。
- 清空标签后拼接结果清空。
- 命中高风险词时触发顶部消息提示。

### 3. 后端/集成测试

验证：

- 提交任务时发出的 `globalPrompt` 为标签拼接结果。
- 老草稿只有文本时仍能正常加载与提交。
- 按钮提示词联动逻辑不回归。
- 自定义提示词模板导入逻辑不回归。

## 实施建议

按以下顺序实现最稳：

1. 先补标签存储服务与 IPC。
2. 再改 `提示词库` 页面。
3. 再改 `套图设计 / 套图生成` 参数区标签化入口。
4. 最后接风险检测与回归测试。

这样可以把数据层、展示层、提交流程分开收敛，减少回归风险。
