# QiuAi 反向提示词底层改造设计

日期：2026-05-06

## 背景

QiuAi 当前在图片生成链路中只支持单一 `prompt` 文本提交。现有上游生图接口不提供原生“反向提示词”字段，因此不能直接依赖 `negativePrompt`、`negative_prompt` 一类接口参数。

用户希望为 `套图设计` 与 `套图生成` 增加独立的“反向提示词”能力，用于稳定约束低质图、不合规图和明显瑕疵图的出现概率，并要求反向模板永久只允许单选一个。

本次设计目标不是为未来接口能力做抽象预留，而是以当前最快速、最稳定的方式落地一个可用的“反向提示词约束”能力：在 QiuAi 内部保存反向提示词模板选择结果，并在最终提交前，将其稳定转换为提示词中的“约束段落”一并发给上游接口。

## 目标

本次改造要实现以下目标：

- 为 `套图设计` 和 `套图生成` 引入独立的反向提示词字段。
- 引入独立的“反向提示词模板库”，与现有正向提示词模板库分离。
- 反向模板永久只允许单选一个，不支持多选叠加。
- 最终请求上游接口时，只发一个 `prompt`，并将反向约束直接并入该 `prompt`。
- 保持旧草稿、旧任务、旧设置兼容，不因缺少新字段而报错。

## 非目标

本次不包含以下内容：

- 不接入 `单图测试` 与 `单图设计` 的反向提示词。
- 不做多模板叠加。
- 不做反向提示词强度调节。
- 不做生成后自动质检与自动重试。
- 不为未来可能存在的原生反向提示词接口做额外抽象。

## 方案选型

采用“单模板选择 + 直接拼接约束段落”方案。

QiuAi 内部保存：

- `negativeTemplateId`
- `negativePrompt`

其中：

- `negativeTemplateId` 表示用户当前选择的反向提示词模板。
- `negativePrompt` 表示本次任务实际使用的反向约束正文。

上游接口调用仍保持单一 `prompt` 字段，不新增伪字段。提交前由统一组装函数将正向提示词与反向约束合成为最终文本。

这样做的好处：

- 前端、草稿、任务、结果展示具备清晰的数据边界。
- 不需要增加额外的接口适配抽象。
- 可以直接围绕当前接口形态落地，改动链路最短。

## 数据设计

### 套图设计草稿字段

`series-design` 新增：

- `negativeTemplateId: string`
- `negativePrompt: string`

### 套图生成草稿字段

`series-generate` 新增：

- `negativeTemplateId: string`
- `negativePrompt: string`

### 反向提示词模板结构

新增独立模板结构，字段与现有正向模板库尽量保持一致：

- `id`
- `name`
- `category`
- `prompt`
- `source`

其中 `source` 取值：

- `system-fixed`
- `custom`

### 默认系统模板

默认内置 3 个系统模板：

1. `电商通用`
2. `电商模特`
3. `电商静物`

模板正文采用用户已确认的默认内容。

## 单选规则

反向模板永久单选，属于硬规则。

系统行为定义如下：

- 同一时刻每个模式只允许存在一个 `negativeTemplateId`。
- 当用户重新选择模板时，新的模板覆盖旧模板。
- 不提供多选 UI。
- 不提供“追加第二个模板”的合并逻辑。

允许用户在模板选定后再补充少量自定义反向词，但本次底层设计先只保证模板单选链路，是否开放补充文本由后续 UI 设计决定。

## Prompt 组装规则

### 设计原则

不将反向模板原文裸拼接到提示词底部，而是统一包装为稳定句式，使模型更容易理解为“约束条件”。

### 最终组装格式

无反向提示词时：

```text
[全局主提示词]
[单图或单项正向提示词]
```

有反向提示词时：

```text
[全局主提示词]
[单图或单项正向提示词]

生成要求：
保留商品主体清晰完整，输出商业电商用途图片。

严格避免以下问题：
[反向提示词正文]
```

### 规范化要求

反向提示词正文在进入最终 prompt 前应做最小规范化：

- 去除首尾空白
- 连续空白压缩
- 中文逗号、英文逗号、顿号混用时做统一
- 去除明显重复项
- 空值不参与拼接

### 结果展示

结果页仍保留 `promptFinal` 作为最终发送文本。后续如需更强可观测性，可再补充：

- `negativePromptUsed`
- `negativeTemplateName`

本次改造不强制新增这两个展示字段。

## 模块边界

### 请求层

请求层负责：

- 不向上游接口发伪字段
- 仅接收已组装好的最终 `prompt`

请求层不负责：

- 模板选择
- 模板解析
- 文本组装

### 生成服务层

生成服务层负责：

- 从草稿中读取 `globalPrompt`、正向 prompt、`negativePrompt`
- 调用统一组装器得到 `promptFinal`
- 将 `promptFinal` 用于远程任务创建

### 工作台与草稿层

工作台与草稿层负责：

- 为 `series-design` 与 `series-generate` 提供默认空字段
- 持久化新字段
- 兼容旧草稿

### 模板库层

模板库层负责：

- 存储独立的反向提示词模板
- 提供默认内置模板
- 支持自定义模板保存与删除
- 保证系统模板不可被删除，但可按后续策略决定是否允许编辑

## 默认值与兼容

为避免旧数据报错，新增字段默认值统一为：

- `negativeTemplateId: ''`
- `negativePrompt: ''`

兼容策略：

- 读取旧草稿时，如果字段不存在，自动补齐为空字符串。
- 读取旧任务或旧工作区快照时，不要求存在反向字段。
- 最终组装时，如果 `negativePrompt` 为空，则完全跳过反向约束段落。

## UI 接入范围

本次底层改造只为后续 UI 接入做准备，明确首批接入范围：

- `套图设计`
- `套图生成`

不进入本次范围：

- `单图测试`
- `单图设计`

这样可以将改动面控制在最核心的批量出图链路上，减少验证成本。

## 测试策略

本次改造测试重点如下：

### 后端服务测试

- 反向模板默认值可读
- 反向模板保存、删除、自定义模板逻辑正常
- `series-design` 在存在反向提示词时能正确组装 `promptFinal`
- `series-generate` 在存在反向提示词时能正确组装 `promptFinal`
- 未选择反向模板时不会多出“严格避免以下问题”段落
- 旧草稿在没有新字段时可正常生成

### 前端源码与状态测试

- `series-design` 草稿包含 `negativeTemplateId` 和 `negativePrompt`
- `series-generate` 草稿包含 `negativeTemplateId` 和 `negativePrompt`
- 选择新的反向模板时会覆盖旧模板，而不是追加

## 风险与控制

### 风险 1：约束过多导致模型输出保守或失败

控制方式：

- 模板永久单选
- 首版不支持多模板叠加
- 统一句式，不做过度强调

### 风险 2：正向与反向内容冲突

控制方式：

- 在后续 UI 中将反向模板与正向提示词分区展示
- 首版不做自动冲突检测，但保留后续扩展空间

### 风险 3：模板正文质量不稳定

控制方式：

- 默认模板先固定为用户确认版本
- 提交前统一规范化和去重

## 结论

本次改造采用“单模板选择 + 直接拼接约束段落”方案是当前最合理路径。

它满足以下要求：

- 当前接口无原生反向提示词也可落地
- 用户永久单选一个反向模板的约束可以被严格执行
- 对现有架构改动最小
- 可以最快接入到现有套图任务链路中

后续实现时，应先完成底层字段、组装器和模板存储服务，再接入 UI。
