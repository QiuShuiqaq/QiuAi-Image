# QiuAi 按图片按组差异化提示词设计

日期：2026-05-07

## 背景

QiuAi 当前的 `套图设计` 与 `套图生成` 都支持多组输出，但每一组实际提交到后端的提示词结构完全一致。

现状：

- `套图设计`：每轮复用同一套 `globalPrompt + imageAssignment.prompt`
- `套图生成`：每轮复用同一套 `globalPrompt + promptAssignment.prompt`
- `batchCount` 仅控制重复次数，不参与提示词差异化

直接结果是：

- 多批次图片之间相似度高
- 用户虽然增加了组数，但没有真正扩大创意搜索空间
- 同一张图在第 1 组和第 2 组之间，往往只是轻微随机差异，不是明确风格差异

本次设计目标是给每一张图片增加“按组差异化提示词”能力，让每一轮真正读取不同提示词，同时尽量复用现有标签库，不再引入第二套词库系统。

## 目标

### 业务目标

- 让 `套图设计` 和 `套图生成` 在多组输出时，不再机械重复同一套提示词
- 支持用户对“同一张图在不同组里”分别定义差异化要求
- 复用现有 `标签库`，降低维护成本和学习成本

### 产品目标

- 差异化配置应当挂在“每张图”上，而不是挂在整个任务顶部
- 用户能清楚看到某张图的第 1 组、第 2 组、第 3 组分别使用什么差异化标签和补充文本
- 保持现有 `全局提示词 -> 单图提示词` 的思路不变，只新增中间一层

### 非目标

- 本次不新增新的标签库体系
- 本次不做自动随机标签推荐
- 本次不做“批量复制当前图片全部组差异化到其他图片”
- 本次不做 AI 自动改写差异化提示词

## 现有问题分析

当前多组相似并不是模型随机性不足，而是输入层面的问题。

以 `套图设计` 为例，当前同一张图的第 1 组、第 2 组通常都走：

1. 全局提示词
2. 这张图的标签拼接
3. 这张图的专属提示词
4. 反向提示词

组与组之间没有任何显式变化项，因此输出仅靠模型的随机扰动变化，差异自然有限。

如果把差异化配置放在任务顶层，例如放在“生成组数”附近，那么默认语义会变成：

- 第 1 组的差异词由所有图片共用
- 第 2 组的差异词由所有图片共用

这种结构不符合当前真实需求，因为不同图片在同一组内希望应用的差异方向并不一致。比如：

- 主图想改“光线、背景、构图”
- 细节图想改“材质、镜头、局部特写”

因此差异化必须以“图片”为主语，而不是以“任务”为主语。

## 方案选型

### 方案 A：任务级组差异化

形式：

- 在 `batchCount` 附近增加 `第1组/第2组/第3组` 差异化输入区
- 整个任务内所有图片共用这一组差异化词

优点：

- 实现简单
- 顶层状态管理容易

缺点：

- 不符合真实业务
- 同组内不同图片无法各自定义差异
- 后续大概率还要重做为图片级方案

结论：不采用。

### 方案 B：按图片、按组差异化

形式：

- 在每个图片卡片内部新增“组差异化”区域
- 每张图都维护自己的第 1 组、第 2 组、第 3 组差异化配置

优点：

- 与业务语义完全一致
- 用户理解直观
- 复用现有标签库的价值最高

缺点：

- 数据结构会比任务级方案复杂
- UI 占用空间变大，需要做折叠或紧凑布局

结论：采用。

## 最终设计

## 一、信息架构

### 套图设计

在每个 `imageAssignment` 图片卡片中，当前结构为：

1. 预览图
2. 图片类型 / 单张比例 / 单张模型
3. 图片专属提示词

调整后变为：

1. 预览图
2. 图片类型 / 单张比例 / 单张模型
3. 组差异化
4. 图片专属提示词

### 套图生成

在每个 `promptAssignment` 卡片中，当前结构为：

1. 图片类型
2. 当前图片提示词

调整后变为：

1. 图片类型
2. 组差异化
3. 当前图片提示词

## 二、组差异化 UI 设计

### 命名

统一命名为：`组差异化`

每组内部显示：

- `第 1 组`
- `第 2 组`
- `第 3 组`

每组包含两部分：

1. `差异化标签`
2. `组补充提示词`

### 标签选择

- 标签数据源复用现有标签库
- 每组可多选标签
- 仅复用标签库数据，不复用“全局已选标签”的状态
- 每组拥有独立选中集

### 组补充提示词

- 单独提供一个小文本框
- 允许用户输入无法用标签表达的补充说明
- 默认可为空

### 展开策略

为了避免每张图卡片过长，采用以下策略：

- 当 `batchCount <= 3`：默认全部展开
- 当 `batchCount > 3`：默认仅展开前 3 组，其余组折叠为紧凑行
- 后续若有需要，再加“展开全部组”入口

本次实现不做复杂的虚拟滚动，保持结构稳定优先。

## 三、数据结构设计

差异化配置不挂在草稿根级，而是挂在每张图自己的配置里。

### 套图设计

在 `imageAssignments[i]` 下新增：

```js
batchVariants: [
  {
    batchIndex: 0,
    selectedTagIds: [],
    promptText: ''
  }
]
```

### 套图生成

在 `promptAssignments[i]` 下新增：

```js
batchVariants: [
  {
    batchIndex: 0,
    selectedTagIds: [],
    promptText: ''
  }
]
```

### 字段说明

- `batchIndex`
  - 对应第几组，从 0 开始
- `selectedTagIds`
  - 当前图片在该组下选中的差异化标签
- `promptText`
  - 当前图片在该组下的自由补充文本

## 四、状态同步规则

### batchCount 增加

当用户把 `batchCount` 从 3 改为 5 时：

- 每张图的 `batchVariants` 自动补齐到 5 项
- 新增项默认为空配置

### batchCount 减少

当用户把 `batchCount` 从 5 改为 2 时：

- 每张图的 `batchVariants` 截断到前 2 项

### 再次增加

再次从 2 改回 5 时：

- 仅恢复前 2 项已有内容
- 第 3 到第 5 项重新按空配置补齐

即：

- 不维护被截断组的历史缓存
- 保持实现简单与数据确定性

## 五、提示词拼接规则

### 套图设计

每次生成某张图在某一组的最终提示词结构：

1. `全局提示词`
2. `该图片在当前组的差异化标签拼接文本`
3. `该图片在当前组的组补充提示词`
4. `该图片原有专属提示词`

即：

```text
globalPrompt
+ batchVariantTagPrompt
+ batchVariantPromptText
+ assignment.prompt
```

### 套图生成

每次生成某条 `promptAssignment` 在某一组的最终提示词结构：

1. `全局提示词`
2. `该条目在当前组的差异化标签拼接文本`
3. `该条目在当前组的组补充提示词`
4. `该条目原有提示词`

即：

```text
globalPrompt
+ batchVariantTagPrompt
+ batchVariantPromptText
+ promptAssignment.prompt
```

### 反向提示词

反向提示词逻辑保持不变，仍然追加在最终正向提示词之后。

## 六、兼容规则

为了兼容老草稿与老任务：

- 若旧数据中没有 `batchVariants`
  - 前端归一化时自动补空数组
  - 提交前再根据 `batchCount` 补齐
- 若旧任务结果中没有该字段
  - 不影响结果展示

因此本次改动必须同步更新：

- 前端 draft 归一化
- 主进程 workspace draft 归一化
- 后端生成 service 的 assignment 归一化

## 七、验证与反馈

### 用户可见反馈

生成完成后，结果展示卡片下方已经有“发送提示词”展示框。

本次应继续沿用该设计，确保这里显示的是：

- 已经拼接好差异化标签
- 已经拼接好组补充提示词
- 已经拼接好单图提示词

这样用户能直接验证：

- 第 1 组和第 2 组是否真的用了不同提示词
- 差异化配置是否生效

### 测试重点

应补充以下回归测试：

1. `batchCount` 变化时 `batchVariants` 自动补齐或截断
2. `套图设计` 生成时，不同组会读取同一张图不同的 `batchVariants`
3. `套图生成` 生成时，不同组会读取同一条目不同的 `batchVariants`
4. 标签库为空、组补充词为空时仍能正常生成
5. 老草稿没有 `batchVariants` 时不会报错

## 八、范围控制

本次实现只覆盖：

- 数据结构扩展
- 参数面板 UI
- 提示词拼接逻辑
- 草稿兼容和测试

明确不包含：

- 批量复制组差异化
- 随机差异化推荐
- 智能生成组差异化提示词
- 按组预设模板导入

## 九、推荐实施顺序

1. 扩展前后端 draft 归一化结构
2. 补测试，先锁定 `batchVariants` 行为
3. 实现 `套图设计` 的图片级组差异化
4. 实现 `套图生成` 的图片级组差异化
5. 更新结果展示中的提示词回显验证

## 结论

本次改造采用“按图片、按组差异化”的设计。

核心原则是：

- 不新建第二套词库
- 复用现有标签库
- 差异化配置挂在每张图自己下面
- 每一组真正读取不同提示词

这样可以最小成本解决“多批次结果过于相似”的问题，同时保持当前产品交互逻辑连续，不会把用户的提示词体系打散。
